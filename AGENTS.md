# courier AGENTS

## 适用范围

- 作用域：仓库根目录及未被子模块 `AGENTS.md` 覆盖的路径
- 目标：约束全局变更边界、执行优先级与接管机制

## 可改范围与禁区

- 可改：与当前任务直接相关的仓库内代码、文档、示例与工具链入口
- 禁区：未授权路径、生产环境操作、破坏性历史改写（如 `reset --hard`、强推）

## 停止条件

- 需求边界不清，且存在多种互斥的接口、目录或执行面方案
- 变更会影响公共 API、生成产物或示例行为，但无法完成最小验证
- 发现仓库事实与既有规则冲突，且无法在当前任务内消解

## 人工接管条件

- 需要仓库外系统权限、敏感配置或网络访问才能继续
- 需要确认公共 API 兼容性、目录拆分方式或新增模块策略
- 需要发布窗口、迁移计划或跨仓库联动调整

## 规则优先级

1. 仓库根 `AGENTS.md`（本文件）
2. 模块 `AGENTS.md`
3. 任务临时约束

## 新规则审校机制

1. 新规则必须附带：目标、边界、风险、回滚方式
2. 启用前必须执行冲突检查（全局 / 模块 / 任务三层）
3. 冲突未消解前不得生效

## 协同约束

1. `README.md` 负责仓库语义与阅读导航，`justfile` 负责稳定命令入口，不在本文件混写 recipe 细节。
2. 新增子模块前必须先确认是否新增，以及采用单入口还是多入口；未确认前不扩展模块级三件套和 root `justfile` 注册。
3. 涉及生成代码时，优先复用现有 `go generate`、root `justfile` 与仓库内生成入口，而不是新增零散脚本。
4. 对公共库 API 的改动默认视为高影响改动；若无法确认兼容性，应先停下并说明风险。

## 实现风格

1. 不做与目标无关的功能扩展、顺手重构或批量风格清理。
2. 优先通过参数传递上下文，避免可变全局状态。
3. 除专有名称外，新增文档、注释与说明尽量使用中文。
4. 避免过度工程、过度防御、过早抽象与无必要注释。

## 依赖与环境

1. 不自行安装依赖、工具或系统组件。
2. 缺依赖导致阻塞时，明确记录为环境前置条件。

## 开发与测试

1. 非极小任务先拆分再执行。
2. 测试描述优先准确中文，直接表达业务语义、输入条件与预期。
3. 无法补测时必须说明缺口与原因。
